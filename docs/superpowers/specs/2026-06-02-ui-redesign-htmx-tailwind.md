# UI Redesign — htmx + Tailwind + Dark Pro Theme

**Date:** 2026-06-02  
**Status:** Approved, ready for planning  
**Replaces:** Current vanilla-JS + hand-written CSS web UI (`agentic_redteam/web/static/app.css`, `app.js`, all templates)

---

## 1. Goal

Replace the plain vanilla-CSS web UI with a polished **Dark Pro** theme using:
- **Tailwind CSS** (83k ⭐) — utility-first CSS, generated by standalone CLI, committed as `static/tailwind.css`
- **htmx 2.x** (37k ⭐) — server-driven partial swaps for wizard navigation + SSE live feed
- **Alpine.js 3.x** (27k ⭐) — minimal client-side interactivity (card selection, toggle, mobile sidebar)

All three committed as static files in `static/` — zero CDN dependency at runtime, works fully offline in the airgapped container.

---

## 2. Visual Design

**Theme: Dark Pro**
- Body background: `bg-gray-950` (`#0f1117`)
- Sidebar background: `bg-gray-900` (`#111827`)
- Card/panel background: `bg-gray-800` / `bg-gray-850` (`#1e2130`)
- Primary accent: red (`red-500` / `#ef4444`) — used for active states, threat indicators, CTAs
- Success: `green-400` (`#4ade80`) — defended results
- Warning: `amber-400` (`#fbbf24`) — errors / sanity flags
- Text: `gray-100` primary, `gray-400` secondary, `gray-600` muted
- Progress bar: gradient `red-500 → orange-500`
- ASR badge: red background, severity-scaled

## 4. Wizard Page (`/`)

### Layout
Two columns inside the content area:
- **Left (180px):** Step list — numbered circles, labels
- **Right (flex-1):** Active step form (htmx swap target)

### Step list states
- **Locked** — gray circle + gray text, not clickable
- **Active** — red filled circle + red text, red left border
- **Completed** — green ✓ circle + white text, clickable (`hx-get="/wizard/step/{n}"`)

### Steps
1. Target LLM — endpoint, model name, API key env var
2. Scope — preset cards OR custom plugin/strategy picker
3. Attacker LLM — optional, shown always (user can leave blank)
4. Judge LLM — endpoint, model name, API key env var
5. App Profile — purpose, tools, roles, data_channels, entities
6. Review & Launch — config summary + run settings (n, concurrency, requested_by, policy_text)

### htmx wiring
- **Next →** button: `hx-post="/wizard/step/{n}/next" hx-target="#step-content" hx-swap="innerHTML"`
- **← Back** button: `hx-get="/wizard/step/{n-1}" hx-target="#step-content" hx-swap="innerHTML"`
- Completed step link: `hx-get="/wizard/step/{n}" hx-target="#step-content" hx-swap="innerHTML"`
- Each partial response includes hidden `<input>` fields carrying all previously entered values forward
- Step 6 (Review) partial: renders a summary + a plain `<form method="post" action="/runs">` submit button that POSTs all accumulated hidden fields

### Alpine.js (step 2 only)
- Preset card click-to-select: `x-on:click` toggles `selected` class, updates hidden `preset` input
- Scope mode toggle (preset vs custom): `x-show` on `#preset-panel` / `#custom-panel`

### New FastAPI routes
```python
GET  /wizard/step/{n: int}        # return step partial (re-render current step)
POST /wizard/step/{n: int}/next   # validate step n fields, return step n+1 partial
```
Both return `HTMLResponse` (Jinja2 partial fragment, no base template wrapper).

Validation in `POST /wizard/step/{n}/next`:
- Step 1: `target_endpoint` and `target_model` non-empty
- Step 4: `judge_endpoint` and `judge_model` non-empty
- Others: no server validation (optional fields)
- On validation failure: re-render same step partial with an inline error message

## 5. Live Run Page (`/runs/{run_id}`)

### Layout
```
Run ID badge + status chip
─────────────────────────────────────
Progress bar (gradient, animated)
─────────────────────────────────────
[Done: 62]  [Succeeded: 21]  [Defended: 38]  [Errors: 3]
─────────────────────────────────────
Live feed (scrolling, new events prepend)
  ⚠  prompt-injection · crescendo · succeeded
  ✓  jailbreak:likert · basic · defended
  …
─────────────────────────────────────
[■ Stop Run]  (bottom right, red button)
```

### SSE wiring (htmx-sse extension)
Container div:
```html
<div hx-ext="sse" sse-connect="/runs/{run_id}/events">
  <div sse-swap="execution_done" hx-swap="afterbegin" id="feed"></div>
  <div sse-swap="run_finished" hx-swap="outerHTML" id="run-status"></div>
</div>
```

SSE server update — add event names and switch data payload from JSON to **pre-rendered HTML fragments**:

```
event: execution_done
data: <div class="feed-row ...">⚠ prompt-injection · crescendo · succeeded</div>

event: run_finished
data: <div id="run-status" class="...">✓ Completed — ASR: 47% <a href="...">View Report →</a></div>
```

htmx-sse swaps the `data:` content directly into the DOM — no client-side JSON parsing. The `_sse()` helper in `app.py` changes to accept pre-rendered HTML instead of a dict. The `run_events` generator calls `render("partials/feed_row.html", ...)` per execution and `render("partials/run_finished.html", ...)` on completion.

The feed row is prepended into `#feed` via `hx-swap="afterbegin"`.

### Run finished state
When `run_finished` fires, `#run-status` swaps to show:
```
✓ Run completed — ASR: 47%
[View Full Report →]   (link to /runs/{id}/report)
```
Stop button disappears, feed freezes.

## 6. Report Page (`/runs/{run_id}/report`)

### Sections
1. **Header** — run ID, status pill, total executions, ASR badge (color-scaled: green <20%, amber 20-50%, red >50%)
2. **Export bar** — "⬇ Download JSON" + "🖨 Print / Save PDF" buttons
3. **Framework Scorecard** — one collapsible `<details>` per framework family (OWASP LLM, OWASP Agentic, MITRE ATLAS, NIST AI RMF, EU AI Act). Each expands to a table: Code | Total | Succeeded | ASR
4. **Plugin × Strategy Heatmap** — table with color-coded ASR cells: `bg-red-900` (>50%), `bg-amber-900` (20-50%), `bg-green-900` (<20%), `bg-gray-800` (0 runs)
5. **Findings** — `<details>` per finding, summary shows plugin/strategy/severity pill, body shows objective + judge rationale + conversation ID
6. **Sanity Flags** — amber callout cards if any flags present

### No htmx on this page — static render, print CSS via Tailwind `print:` variants

## 9. Backend Changes Summary

### app.py additions
```python
@app.get("/wizard/step/{n}")
async def wizard_step(n: int, request: Request):
    # Re-render step n partial (for Back navigation / completed-step edit)
    ...

@app.post("/wizard/step/{n}/next")
async def wizard_step_next(n: int, request: Request):
    # Validate step n, return step n+1 partial (or step n with error)
    ...

### SSE generator update
Add `event:` lines before each `data:` line. For `execution_done`, return a pre-rendered HTML row fragment instead of raw JSON so htmx can swap it directly. Requires two new partials: `partials/feed_row.html` and `partials/run_finished.html`.

### Template additions
```

### Existing routes: no changes to signatures or logic — only templates change.

## 11. Decisions Log

| Decision | Choice | Reason |
|----------|--------|--------|
| CSS framework | Tailwind CSS via standalone CLI | 83k ⭐, no Node.js, committed artifact |
| JS framework | htmx 2.x + Alpine.js 3.x | htmx 37k ⭐ + Alpine 27k ⭐; htmx for server-swaps/SSE, Alpine for 3 client-side interactions |
| CDN vs local | All local (committed to static/) | Airgapped container environment |
| Wizard state | Hidden input accumulation | No session dependency; pure HTML form |
| SSE format | Named events + pre-rendered HTML fragments | htmx-sse requires named events; fragments avoid client-side JSON parsing |
| Report page | Separate page from live run | User-confirmed preference |
| Theme | Dark Pro (dark bg, red/amber accents) | User-selected |
| Layout | Fixed sidebar + scrollable main | User-selected |
| Wizard style | Step list left + content right | User-selected |
| Live run style | Stats cards + live feed | User-selected |
